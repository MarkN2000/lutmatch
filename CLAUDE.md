# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 基本ルール

- **ユーザーへの返答は必ず日本語で行うこと。**
- **DRY 原則・SOLID 原則**を意識してコードを設計・実装すること。
- **サブエージェントを積極的に活用すること。** メインエージェントの役割は「設計・タスク分解・サブエージェントへの委譲・成果物の監査・ユーザーへの最終報告」である。できるだけ目的に応じたサブエージェントに委譲し、上がってきた成果物を監査してから報告する。トークンを消費する実作業（ファイルの読み込み・検索・実装・テスト実行など）はメインで抱え込まず、原則サブエージェントへ委譲する。必要に応じて並列で実行する。
- **サブエージェントのモデルがメイン (Fable) にフォールバックする起動は原則禁止する。** 未指定で起動するとメインのモデルを継承し、利用制限を浪費してしまうため。
  - サブエージェントは起動時に必ず `model` を明示すること。定型的な調査・検索・機械的な作業は `sonnet`、数値計算のコアや難しいデバッグなど高難度タスクは `opus`（Opus 4.8）。
  - `fork`（subagent_type）はモデル指定を無視して Fable を継承するため、opus でうまく解決できなかった高難易度の重要なタスクのみに使う。
  - 新しいエージェント定義を `.claude/agents/` に追加するときは、frontmatter に必ず `model` を書くこと。

## プロジェクト概要

**LUT Match** — 元画像（Source）と参考画像（Reference）の2枚から、色を近づける 3D LUT を自動生成し `.cube` として書き出すWebアプリ。完全ブラウザ内処理（サーバー送信なし）。

**現状は実装済み・リリース前**：`src/` に全機能（コア・Worker・WebGL プレビュー・UI・i18n）、`tests/` に vitest テスト、`.github/workflows/ci.yml` に CI が揃っている。実装に関するすべての判断は `spec.md` を唯一の情報源とすること。仕様と実装が食い違う場合は仕様を確認し、仕様変更が必要なら `spec.md` の決定事項ログ（§13）を更新する。

ドキュメント・UI文言・コミュニケーションは日本語が基本（UIは日英 i18n あり）。

## コマンド

スタックは Vite + TypeScript（フレームワークなし・Vanilla TS）、テストは vitest：

- 開発サーバー: `npm run dev`
- ビルド: `npm run build`（`vite build` → `dist/`、Cloudflare Pages にデプロイ）
- 型チェック: `tsc --noEmit`（CI でも実行）
- テスト: `npx vitest run`／単一ファイル: `npx vitest run tests/<file>.test.ts`

外部ランタイム依存（CDN含む）はゼロを維持する。依存は原則 devDependencies（vite / typescript / vitest）のみ。ただしテスト専用の dev 依存（現在 `lzma`。Resonite Bmp3D の LZMA 往復検証に使用）は許可する（ランタイム＝`src/` へは持ち込まない）。

## アーキテクチャ（spec.md §8 のディレクトリ案）

- `src/core/` — 純粋なアルゴリズム層（UI非依存、テスト対象）
  - `colorspace.ts`（sRGB↔リニア、Lab）→ `mkl.ts`（線形マッチ）／`histmatch.ts`（ヒストグラムマッチ）→ `pipeline.ts`（モード合成・後段処理）→ `lut.ts`（格子生成・平滑化）→ `cube.ts`（シリアライズ）
  - `src/core/resonite/` — Resonite `.resonitepackage` 書き出し（`lzma.ts`＝Bmp3D 用 LZMA／`brotliStore.ts`＝FrDT 用 store-only Brotli／`bmp3d.ts`＝3D LUT アセット／`zip.ts`＝最小 ZIP／`package.ts`＝テンプレート差し替え組み立て。すべて純粋・fetch/DOM 非依存）
- `src/worker/match.worker.ts` — 重い計算は Web Worker で実行（メインスレッドをブロックしない）
- `src/gl/preview.ts` — WebGL プレビュー＋Canvas 2D フォールバック（Safari 対策で必須維持）
- `src/ui/`, `src/main.ts` — 素の DOM + CSS、`src/i18n/{ja,en}.ts` — 文言辞書
- `tests/` — vitest（core/ 対象）、`public/` — サンプル画像

自動マッチは3モード：A=MKL のみ / B=HM のみ / C=HM→MKL→HM 複合（既定）。変換関数を N³ 格子（既定33）に適用して LUT 化する。

## 実装時に守るべきアルゴリズム上の決定（spec.md §5・§13）

これらはレビューで確定済みの決定事項。変更する場合は必ず理由とともに §13 に記録する。

- 統計計算は**リニア化した RGB** 上で行う（sRGB デガンマ → 計算 → 再ガンマ）
- HM のビン割りは**ガンマ（sRGB）空間で等間隔**（リニア空間だとシャドウ階調が破綻するため）。HM カーブは**残差平滑化**（恒等 y=x を引いた残差に σ=2 ビンのガウシアン・`HM_RESIDUAL_SMOOTH_SIGMA`）を掛けて暗部の傾きスパイク＝スペックルを抑制。恒等残差は不変で同一性は保持、平滑化後に単調化を再保証（§5.3）
- 複合パイプラインの**中間段では 0–1 クランプしない**。クランプは最終ステップのみ（範囲外値のガンマ⇄リニア変換は符号付き `sign(x)·γ(|x|)`、HM カーブは端点傾きで線形外挿）
- 外挿域は Source 分布の**マハラノビス距離**（既定 d0=3.0）に基づき Identity 方向へ減衰（距離は**格子点をリニア化**して Source のリニア空間統計で測る）
- **強度（Identity ミックス）は自動マッチ結果のみに適用**し、手動調整はフル効果。処理順序：格子適用 → 外挿減衰 → 平滑化 → **Identity ミックス（強度）** → **手動調整（露出→色温度/ティント→コントラスト→彩度）** → **0–1 クランプ**
- MKL は退化対策必須：共分散正則化 `Σ+εI`・固有値 0 クランプ、ランク落ち/有効画素 < `N_MIN_PIXELS`（1024）時は**平均シフトのみへフォールバック**＋UI 警告
- `.cube` の TITLE は印字可能 ASCII にサニタイズ（日本語不可。ファイル名自体は日本語可）
- 解析解像度は長辺 512px（内部定数 `ANALYSIS_LONG_EDGE`）、アルファ 50% 未満の画素は統計から除外

## テスト方針（spec.md §11）

- ユニット：色空間往復誤差、MKL 閉形式解、HM の単調性、.cube フォーマット妥当性
- ゴールデンテスト：固定入力ペアの LUT スナップショット比較（回帰検知）
- 同一性テスト：Source＝Reference のとき LUT ≒ Identity になること
- **ゴールデンの更新**：意図的なアルゴリズム変更でスナップショットが変わる場合のみ `npx vitest run -u` で更新する。更新時は変化が改善方向であることを確認し、理由をコミットメッセージ（必要なら spec §13）に記録すること

## UI の最重要原則（spec.md §6）

- 1画面構成（ウィザード化しない）。パラメーター操作中、プレビューが常に画面内に見えること（sticky）
- 常時表示はモード3択＋強度スライダーのみ。詳細7項目はアコーディオン（既定閉）
- モバイルは一級対応（比較スライダーはハンドルドラッグのみ、タッチターゲット44px以上）

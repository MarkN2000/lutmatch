# 仕様書：参考画像ベース LUT ジェネレーター（仮称：LUT Match）

- 版数: v0.2
- 最終更新: 2026-07-12
- ステータス: 確定

---

## 1. 概要

### 1.1 目的

「元画像（Source）」と「参考画像（Reference）」の2枚を入力すると、元画像の色を参考画像の雰囲気に近づける 3D LUT を自動生成し、`.cube` ファイルとして書き出せる Web アプリを開発する。

例：デジタルカメラで普通に撮った写真（Source）と、昔フィルムカメラで撮った写真（Reference）を入力すると、フィルム風の色合いになる LUT が得られる。

### 1.2 コンセプト

- **完全ローカル処理**：画像はサーバーに一切送信しない。すべてブラウザ内（JavaScript / Web Worker）で計算する
- **自動＋手動**：自動カラーマッチで 8 割を作り、手動パラメーターで追い込む
- **汎用出力**：生成した LUT は DaVinci Resolve / Premiere Pro / Photoshop / Lightroom（プロファイル経由）/ OBS など `.cube` 対応ソフトでそのまま使える

### 1.3 スコープ外（v1 では実装しない）

- 動画ファイルの入力
- グレイン・ビネット等の空間エフェクト（LUT は色の写像のみで原理的に含められないため）
- サーバーサイド処理、アカウント機能、課金
- プリセット保存、PNG HALD 出力（将来検討 → §12）

## 2. 用語

| 用語 | 意味 |
|---|---|
| Source | LUT を適用したい元画像 |
| Reference | 目標とする雰囲気の参考画像 |
| 3D LUT | RGB 入力色 → 出力色の 3 次元ルックアップテーブル |
| `.cube` | Adobe/IWLTBAP 系で標準的な LUT テキスト形式 |
| MKL | Monge-Kantorovich 線形化。線形変換による色分布マッチ |
| HM | ヒストグラムマッチング。チャンネル毎の累積分布を一致させる非線形マッチ |
| Identity LUT | 入力＝出力の無変換 LUT。強度ミックスの基準に使う |

## 4. 機能要件

### 4.1 画像入力

- **Source は必須、Reference は任意**。Reference 未投入時は自動カラーマッチを行わず、**恒等変換を基底**とした手動 LUT 作成モードになる（専用のモード切り替え UI は用意せず、Reference の有無がそのままモードを決める。詳細は §4.3/§6.2）
- **Source の入力口はプレビュー領域**：空状態ではプレビュー全面がドロップゾーンとなり、ドラッグ＆ドロップとクリック（タップ）でのファイル選択の両方を受け付ける。投入後もプレビュー領域は常時ドラッグ＆ドロップを受け付け、ドロップで Source を差し替える（クリックでのファイル選択は誤操作防止のため空状態限定）。明示的な操作用にプレビュータブバー右端へ「差し替え」「削除（×）」ボタンを置く（§6.1）
- **Reference の入力口は「色合わせ」アコーディオン内のドロップゾーン**（§6.1）：ドラッグ＆ドロップとクリック選択の両対応で、サムネイル・クリック差し替え・×（削除）ボタンを備える
- ページ全体で `dragover` / `drop` の既定動作を抑止する（入力口の外へ誤ってドロップした際に、ブラウザが画像ファイルを開いて作業状態が失われるのを防ぐ）
- Source / Reference とも投入済み画像を後から削除できる（Source はプレビュータブバーの ×、Reference はドロップゾーンの ×）。削除時は世代カウンタをインクリメントし、進行中のデコード処理を無効化する（デコード完了時に古い世代の結果を破棄し、削除後の状態を上書きしない）
- 対応形式：JPEG / PNG / WebP（ブラウザがデコードできるもの）
- **デコード経路**：`createImageBitmap(file, { imageOrientation: 'from-image', resizeWidth, resizeHeight })` を用いて、目的の解像度へ縮小したうえで `ImageBitmap` を得る。**フル解像度を一度も canvas に描画しない**（iOS Safari の canvas 面積上限〔約 16.7MP〕を超えて描画が失敗するのを避けるため）。`imageOrientation: 'from-image'` により EXIF 回転はデコード時に適用される
- 入力画像は解析用に長辺 **512px** へ縮小して統計計算に使用（プレビュー表示用は長辺 2048px まで別途保持）。サムネイル表示も同一のデコード経路（`createImageBitmap` + `resizeWidth/resizeHeight`）を使い回し、回転ズレやオリエンテーション不一致を防ぐ
  - 根拠：MKL（平均・共分散）は数千 px、HM（256 ビン CDF）は長辺 256px 相当で概ね足りるが、シャドウ・ハイライトの分布の裾の推定安定性のため余裕を持たせる。処理時間の支配項は元画像のデコード/リサイズであり、512px の統計計算コストは数十 ms と無視できる。内部定数 `ANALYSIS_LONG_EDGE` として変更可能にする
- 透過 PNG はアルファ 50% 未満の画素を統計計算から除外する
- 広色域画像（Display P3 等）は Canvas 経由で sRGB に変換して扱う（制限としてヘルプに明記）。canvas / `ImageData` の `colorSpace` は既定（`'srgb'`）を明示指定して扱う
- **巨大画像の扱い**：`>50MP` を事前検出してエラーにする方式は廃止する（`createImageBitmap` でフル解像度を描画しないため事前サイズ判定の必要がなくなる）。代わりにデコード失敗・メモリ不足（`createImageBitmap` の reject 等）を捕捉し、トーストで通知して直前の状態を維持する
- **HEIC 等の非対応形式**：ブラウザがデコードできない形式はデコード失敗として捕捉し、エラー文言に「JPEG に変換してください」等の具体的な案内を含める
- 画像差し替え時は直前の `ImageBitmap` を `ImageBitmap.close()`、生成済み Object URL を `URL.revokeObjectURL()` で明示解放してから再計算する

### 4.2 自動カラーマッチ（3 モード）

| モード | UI 表示名（日/英） | 内部処理 | 想定用途 |
|---|---|---|---|
| A | ナチュラル / Natural | MKL のみ | 破綻を避けたい・被写体が大きく異なる場合 |
| B | 忠実 / Faithful | HM のみ（チャンネル別） | トーンカーブごと強く寄せたい場合 |
| C（既定） | バランス / Balanced | HM → MKL → HM の複合パイプライン | 通常はこれ。非線形の再現と安定性の両立 |

- **UI 上のグループ見出し**は日本語「**色合わせ**」・英語 "Color Match"（グループ内に Reference ドロップゾーンを含むため、見出し自体が「参考画像に色を合わせる」機能を示す短い名称とする。本仕様書ではアルゴリズム名として引き続き「自動マッチ」と表記する。§13）
- **Reference 未投入時は自動マッチ自体が無効**になる（本表のモード切り替え・強度・ノイズ抑制など自動マッチ系コントロールは Source・Reference 両方投入時のみ有効。有効条件の詳細は §4.4）
- モード変更時は自動で再計算（Web Worker、目標 1 秒以内）
- 計算は知覚均等性のため **リニア化した RGB** 上で行う（sRGB EOTF でデガンマ → 計算 → 再ガンマ）。Lab 空間案との比較は検討の結果不採用で確定（§13。HM のみガンマ空間で動く意図的な例外 → §5.3）

### 4.3 LUT 生成

**Reference 未投入時の分岐**：Reference がない場合、自動マッチ（本節の格子適用・外挿減衰・平滑化・強度ミックスをすべて含む）を丸ごとスキップし、厳密な恒等変換を基底として手動調整→色相カーブ→残差カーブ加算→クランプのみを適用する（§5.5 の手順のうち 1〜4 を省略し手順 5 から開始）。この恒等経路は「平滑化等が no-op だから省略される」のではなく、**純恒等＋手動＋カーブと定義する設計判断**である（経緯は §13）。

以下は Reference 投入時（自動マッチ経路）の手順である：

1. §4.2 の変換関数 f(r,g,b) を得る（モード B/C では HM カーブ構築に**ノイズ抑制**スライダー連動の傾きクランプ・残差平滑化パラメータ化が入る。詳細は §5.3）
2. N×N×N（既定 33）の格子点すべてに f を適用して 3D LUT 化
3. **範囲外挙動**：入力画像に存在しない色域（統計の外挿域）は Identity 方向へ減衰させ、極端な色化けを防ぐ
4. **平滑化**：3D ガウシアンぼかし（**スムージングスライダー連動**）でバンディング・色飛びを抑制

処理の詳細な順序は §5.5 に従う（外挿減衰 → 平滑化 → 強度ミックス → 手動調整 → 色相カーブ → 残差カーブ加算 → クランプ）。

### 4.4 手動調整パラメーター

生成された LUT に対して、以下を LUT に「焼き込む」形で追加調整できる。すべてリアルタイムでプレビューに反映。

| パラメーター | 範囲 | 既定値 | 備考 |
|---|---|---|---|
| 強度（ミックス） | 0–100% | 80% | **自動マッチ結果と Identity のブレンド率（手動調整には影響しない）** |
| スムージング | 0–100 | 20 | §4.3 の平滑化強度 |
| ノイズ抑制 | 0–100 | 0 | HM カーブの傾き急変を抑制し、ノイズの増幅を防ぐ（ノイズ除去そのものではない）。モード A（ナチュラル）では無効。§5.3 |
| 露出 | −2.0〜+2.0 EV | 0 | |
| コントラスト | −50〜+50 | 0 | 中間グレー基準の S カーブ |
| 彩度 | −100〜+100 | 0 | |
| 色温度 | −100〜+100 | 0 | Lab の b* 軸相当 |
| ティント | −100〜+100 | 0 | Lab の a* 軸相当 |
| ブラック保護 | 0–20% | 5% | 指定した**知覚（sRGB）輝度**以下を統計マッチから除外し、締まった黒を維持。内部では sRGB→リニア変換した値でリニア輝度と比較する |

- **有効条件**：露出・コントラスト・彩度・色温度・ティント（本表の手動系 5 種）およびカーブ編集（§5.7・§5.8）は **Source のみで有効**（Reference 不要）。強度・スムージング・ノイズ抑制・モード切り替えなどの自動マッチ系コントロールは **Source と Reference の両方が揃って初めて有効**になり、Reference 未投入時は UI 上で無効化のうえ理由をツールチップで表示する
- **ブラック保護と恒等経路**：Reference 未投入時（恒等経路、§4.3）は自動マッチの統計自体が存在しないため、ブラック保護しきい値は UI 設定によらず **0 として扱う**（無効化されたスライダーの既定値 5% が、ヒストグラム・実効カーブの表示ドメインを黙って切り詰めてしまうのを防ぐため）
- カーブ編集（§5.7・§5.8）は本表のスライダー群とは別クラスの編集面（マスター／R／G／B／色相／彩度 6 チャンネル分のコントロールポイント集合）であり、LUT へは独立の残差として加算される。マスター／R／G／B（§5.7）は格子入力座標基準、色相／彩度（§5.8）は base 格子の色相 h 基準で、いずれもクランプ前に加算される
- リセットは 3 段階のスコープを持つ（詳細は §6.2）：①「カーブ」パネル内の「◯◯をリセット」（選択中チャンネルのみ・動的ラベル）／「すべてのカーブをリセット」（カーブ全体）、②「詳細調整」アコーディオン内の「詳細調整をリセット」（詳細 7 項目のみ）、③左レール末尾の「すべてリセット」（強度・ノイズ抑制・詳細調整・カーブ編集をまとめて初期化。自動マッチ結果は保持）

### 4.5 プレビュー

- Source に現在の LUT（自動＋手動込み）を適用した結果を常時表示
- **ビフォーアフター比較スライダー**：1 枚の画像の上で境界線をドラッグして適用前/後を左右比較（初期位置 50%）
- 表示切替タブ：`元画像` / `適用後` / `比較スライダー`（既定）
- Reference は **投入時のみ** サムネイルとして参照表示する（Reference 未投入時は小窓を含め非表示。§6.1 も同様）
- 画像は常にステージへフィット表示する（拡大縮小・パン操作はなし）
- **描画経路**：WebGL2 の 3D テクスチャに LUT を格納する経路を主経路とする（2D パック LUT 案は採用しない）。`webglcontextlost` / `webglcontextrestored` をハンドリングし、コンテキスト喪失時は復旧を試み、復旧に失敗した場合は Canvas 2D フォールバックへ自動切替する（詳細は §7）
- **バッキングストア解像度**：`devicePixelRatio` を上限 2 でクランプして決定する（高 DPI 端末での過大な描画コスト・メモリ消費を避ける）

### 4.6 書き出し

- 形式：`.cube`（3D LUT）
- サイズ選択：**17 / 33 / 65**（既定 33）
- 出力内容＝自動マッチ＋手動調整をすべて合成した最終 LUT。**Reference 未投入時（手動 LUT 作成モード）は、恒等変換を基底に手動調整とカーブ編集を合成した LUT**（恒等基底＋手動調整＋カーブ編集）を出力する
- ファイル名：ユーザー入力（既定 `lutmatch_YYYYMMDD_HHmm.cube`）。名前は `TITLE` ヘッダーにも埋め込む
- 併せて「適用後画像を PNG 保存」ボタンも用意（確認用）。保存はプレビュー解像度（長辺 **2048px**）で行う

#### 4.6.1 Resonite 用書き出し

- 「Resonite 用に保存」ボタン（en: "Save for Resonite"）を書き出しバーに追加する。押すと現在の LUT を `.resonitepackage`（内部は ZIP）として生成しダウンロードする。完全ブラウザ内処理
- **テンプレート差し替え方式**：実機インポート検証済みのテンプレートパッケージ（`public/resonite-template/`）を素材とし、LUT アセット（**Bmp3D**＝LZMA 圧縮の 3D テクスチャ）とそれを参照するメインオブジェクト（**FrDT**＝store-only Brotli）のみを現在の LUT に応じて作り直し、参照ハッシュ（SHA-256）を書き換えて再パッケージする。フォント・シェーダ等の他 8 アセットとサムネイル（Metadata の .bitmap）はテンプレートのまま流用する
- 手順：①`state.currentLut` を `encodeBmp3d(grid, N)` で新 LUT アセット化・SHA-256 算出 → ②FrDT 展開済みペイロード内の旧 LUT ハッシュ ASCII を新ハッシュへ同一長バイト置換（出現 1 件でなければ破損として中断）→ ③`brotliStore()` で圧縮し 9 バイトの FrDT ヘッダー（`"FrDT"`+0x00×4+0x03）を付与・SHA-256 算出 → ④record JSON の `assetUri`（メイン FrDT 参照）・`assetManifest`（旧 LUT エントリの hash/bytes）・`name` を更新 → ⑤`R-Main.record` + `Assets/<sha256>`×10 + `Metadata/<hash>.bitmap` を ZIP 化
- **N は書き出しバーで選択中のサイズに追従**する（17 / 33 / 65）
- ファイル名は `.cube` のベース名に連動（`foo.cube` → `foo.resonitepackage`）、record の `name` も同ベース名にする
- ZIP は `CompressionStream('deflate-raw')` が使える環境では DEFLATE、無ければ STORE に自動フォールバック（DEFLATE 結果が元より大きい場合も STORE を採用）
- コア層（`src/core/resonite/`）は純粋・fetch/DOM 非依存でユニットテスト対象。テンプレートのフェッチ・ダウンロードトリガーは `src/main.ts` が担う

## 5. アルゴリズム仕様

### 5.1 前処理

1. 画像を長辺 512px に縮小、ピクセル配列を取得
2. sRGB → リニア RGB へ変換
3. ブラック保護しきい値未満の画素を統計対象から除外（UI の知覚 % を sRGB→リニア変換した値を Rec.709 **リニア輝度**と比較。Source / Reference 両方に対称に適用する）
4. **有効画素数の保護**：アルファ除外（§4.1）とブラック保護を適用した後の有効画素数が `N_MIN_PIXELS`（既定 **1024**）未満の場合、統計推定が不安定になるため §5.2 と同様に**平均シフトのみのフォールバック**へ切り替え、UI に警告を表示する

### 5.2 MKL（モード A）

Source / Reference それぞれの平均ベクトル μ と共分散行列 Σ を計算し、閉形式解

T = Σs^(-1/2) · (Σs^(1/2) · Σr · Σs^(1/2))^(1/2) · Σs^(-1/2)

により線形写像 `f(x) = T(x − μs) + μr` を得る（Pitié & Kokaram 2007）。3×3 の固有値分解のみで済むため軽量。

**退化・破綻対策**：実画像の**異方的な色分布**では `Σs^(-1/2)` の小固有値方向のゲインが爆発し、MKL は 3×3 でチャンネルを混ぜるため微小な粒状ノイズを彩度・色相ノイズ（マゼンタ/緑のスペックル、青・赤の過強調）へ増幅する。これを次の 3 段で抑える。

- **収縮正則化（絶対下限つき）**：共分散を正則化する `Σ ← Σ + εI`、`ε = max(trace(Σ)/3 × 1e-4, MKL_EPS_ABS)`。相対量 `trace/3 × 1e-4` だけでは暗部（低分散）画像で `ε≒0` となり無力なため、8bit 量子化ノイズ相当のリニア分散を**絶対下限** `MKL_EPS_ABS`（既定 **1e-5**、σ≒0.0032 リニア）として課す。Source・Reference の両共分散へ同一に適用し、行列平方根・逆行列を数値的に安定させる。固有値は `sqrt` を取る前に **0 でクランプ**する
- **ゲイン上限クランプ**：輸送行列 T は対称正定値。構築後に固有値分解し、方向別ゲイン（固有値）を `[1/MKL_MAX_GAIN, MKL_MAX_GAIN]`（既定 `MKL_MAX_GAIN = **4.0**`）にクランプして再構成する。上限 4.0 はリニア空間で 2 EV（4×）に相当し、露出・コントラスト差の**正当な全域スケーリングは素通しつつ**、それを超える病的な方向ゲインだけを頭打ちにする。T が数値誤差で厳密対称でない場合は `(T+Tᵀ)/2` で対称化してから分解する。Source=Reference のとき T=I なので同一性は保たれる
- **段階的フォールバック（異方性ブレンド）**：Source 共分散の異方性比 `r = λmin/λmax` に応じ、フル MKL 行列 T と**チャンネル別対角スケール** `D = diag(σr_i/σs_i)`（同じくゲインクランプ・チャンネルを混ぜないため色相ノイズを増幅しない）の間を重み `w` で補間する（`T' = w·T + (1−w)·D`）。`w` は `r ≥ MKL_ANISO_FULL`（既定 **1e-2**）で 1、`r ≤ MKL_ANISO_DIAG`（既定 **1e-4**）で 0、その間は log 空間 smoothstep。これによりニュートラルな Reference やグレースケール Source でも**平均シフトへ即崩壊せず**対角スケールで穏当にマッチする（旧・二値ランク判定 `MKL_RANK_RATIO` は廃止）
- **完全退化のフォールバック**：有効画素数が `N_MIN_PIXELS`（既定 **1024**）未満の場合のみ、線形写像を求めず**平均シフトへフォールバック**する：`T = I`、`f(x) = x − μs + μr`。この場合は UI に警告を表示する

### 5.3 HM（モード B）

R/G/B 各チャンネルについて Source の累積分布関数を Reference の累積分布関数へ写像（256 bin、単調性を保証）。得られる 3 本の 1D カーブを 3D 格子へ適用する。

**ビン割りはガンマ（sRGB）空間で等間隔に定義する。** リニア空間で等間隔ビンを切るとシャドウ側の階調が極端に粗くなり、フィルムルックで重要な暗部の再現が破綻するため。リニア値はビン参照時にガンマ空間へ写像して扱う。

**HM はガンマ（sRGB）空間で動く 1D 単調リマップである。** すなわち HM は「統計計算はリニア RGB 上で行う」原則（§4.2）の**意図的な例外**であり、CDF の構築とカーブ適用はガンマ空間で行う（この方が暗部階調が保たれるため）。この例外は仕様上の確定事項として扱う。

**逆 CDF の求め方**：Reference の逆 CDF は分位点の**線形補間**で求める。得られる写像カーブは**単調非減少であることを保証**する（数値誤差で非単調が生じた場合は単調になるよう補正する）。

**残差平滑化（暗部スペックル対策）**：暗部（リニア輝度 5〜15% 程度）ではソースのヒストグラムが疎になりやすく、逆 CDF 由来のリマップカーブが**単一ビンで急峻に跳ねる傾きスパイク**を作る。このスパイクはソースの微小な粒状ノイズ（±数レベル）を、チャンネル間で不整合な方向の大きな色差へ増幅し、暗部にマゼンタ/緑のスペックルを生む（実機検証で確認）。対策として、カーブから恒等 `y=x` を引いた**残差**にごく弱い 1D ガウシアン（σ=2 ビン、内部定数 `HM_RESIDUAL_SMOOTH_SIGMA`）を掛け、恒等を足し戻してスパイクを近傍ビンへ均す。恒等入力（残差=0）は平滑化後も厳密に恒等のままで**同一性（Source=Reference → LUT≒Identity）を壊さない**。平滑化後に単調化を再適用して単調非減少を保証する。σ を上げるほどスペックルは減るが Reference 分布への追従がわずかに甘くなるトレードオフがあり、σ=2 は暗部の傾きスパイクを 1/5 程度・ギザつき（2 階差分）を 1/19 程度に抑えつつ CDF マッチの平均挙動をほぼ保つ控えめな値。

**ノイズ抑制スライダー（0–100、既定 0）**：3D LUT は画素単位の色変換であり、空間的なノイズ除去は原理的に不可能である。本機能は「変換の勾配（傾き）を抑えてノイズを増幅しない」アプローチであり、ノイズ除去そのものではない。上記の残差平滑化は固定 σ=2 だったが、色境界での色ノイズ（スペックル）が目立つ画像向けに、ユーザーが抑制強度を調整できるようにする。1 本のスライダー値 s は、次の 2 つの機構を**同時に**駆動する（両者は相補的：ガウシアン平滑化は幅の狭い傾きスパイクに、傾き上限クランプは幅の広い急峻区間に効く）。

1. **残差平滑化 σ のパラメータ化**：`σ(s) = HM_RESIDUAL_SMOOTH_SIGMA × 2^(s/25)`（s=0 で σ=2 ビン＝現行の固定値、s=100 で σ=32 ビン）
2. **カーブ傾き上限クランプ**：`S_max(s) = 2^(4(1−s/100))`（s=0 で 16 ≒ 実質無効、s=100 で 1＝恒等＋オフセットへ漸近）。**s=0 ではクランプ処理自体を完全スキップ**し、現行挙動（σ=2・クランプなし）と完全に一致させる

上記の係数マッピングは出発点であり、実装時に実画像ペアで校正する。

**傾きクランプの実装（CLAHE 式再分配）**：HM カーブの微分（ビンごとの傾き `d[i] ≥ 0`）を上限 `S_max` でクリップし、超過分を全ビンへ均等再分配する（CLAHE のクリップリミット再分配と同じ構造。2〜3 回の反復で収束）。`d ≥ 0` を保ったまま操作するため単調性は自動的に保たれ、端点（カーブの総上昇量）も保存される。

**カーブ構築の処理順**：CDF 写像 → 単調化 → 傾きクランプ（CLAHE 式再分配）→ 残差平滑化 → 単調化再保証。クランプを先に行うことで生じる折れ点を後段の残差平滑化が丸め、かつ正規化ガウシアンは微分の最大値を増やさないため、平滑化後も傾き上限の保証が維持される。

**作用範囲**：モード B は HM 1 段、モード C は HM→MKL→HM の 1 段目・2 段目双方に同一パラメータを適用する。モード A（MKL のみ）は HM を使わないため本スライダーは無効（UI はグレーアウトし理由をツールチップ表示）。強度（Identity ミックス）・スムージング（§5.5 の 3D 格子平滑化）・手動調整とは独立したパラメータである。

**実装インターフェース**：`GenerateLutOptions` に `noiseSuppression: number`（0–100）を追加し、UI → Worker → `buildMatchTransform` → `buildHistMatch`（引数追加）の経路で伝搬する設計とする。

### 5.4 複合（モード C・既定）

`HM → MKL → HM` の直列適用（color-matcher の HM-MVGD-HM 構成に準拠、線形段は MKL を使用）。各段は §5.2 / §5.3 の実装を再利用する。

**段間データフロー**：解析画素集合（§5.1 の有効画素サンプル）を各段の変換で順に押し出し、**次段の統計（CDF・平均・共分散）は変換後のサンプルから推定する**。具体的には：

1. **1 段目 HM**：元の Source 分布 → 元の Reference 分布へマッチ。Source サンプルをこのカーブで変換し、中間サンプル S₁ を得る
2. **MKL 段**：S₁ の平均・共分散を Reference の平均・共分散へ合わせる線形写像を求め、S₁ を変換して S₂ を得る
3. **2 段目 HM**：**「HM→MKL 適用後の Source 分布」（= S₂ の分布）を「元の Reference 分布」へマッチ**させる。すなわち S₂ の CDF を元 Reference の CDF へ写像する

**中間段では 0–1 クランプを行わない**（範囲外値を保持したまま次段へ渡す）。段ごとにクランプすると情報が失われ、複合の利点が損なわれるため。クランプは §5.5 の最終ステップのみ。

**範囲外値（負値・>1）の扱い**：中間クランプなしで生じる範囲外値を、段間の色空間変換で破綻させないため次の規約を用いる：

- **ガンマ ⇄ リニア変換**：符号付き拡張 `sign(x)·γ(|x|)` を用いる（負値でも符号を保ったまま変換）
- **HM カーブ**：ビン範囲 `[0,1]` の外側では、端点の傾きで**線形外挿**する

### 5.5 LUT 化と後段

処理順序は以下のとおり（**強度ミックスは自動マッチ結果のみに作用し、手動調整・カーブ編集（RGB・色相）はそのフル効果が保たれる**点に注意）：

1. **N³ 格子点に f を適用**
2. **外挿域の Identity 減衰**：Source 色分布の平均・共分散に基づく**マハラノビス距離**がしきい値 d0 を超える格子点は、距離に応じて f の出力を Identity 方向へ線形に減衰させる（d0 と減衰幅は実装時にチューニング、既定 d0=3.0）。マハラノビス距離は、**格子点座標をリニア化したうえで** Source の**リニア空間統計**（μs, Σs）に対して測る（`d² = (x_lin − μs)ᵀ Σs⁻¹ (x_lin − μs)`）
3. **3D ガウシアン平滑化**（スムージングスライダー連動）
4. **Identity ミックス（強度）**：ここまでの自動マッチ結果を Identity とブレンドする。強度はこの段のみに作用し、以降の手動調整・カーブ編集には影響しない
5. **手動調整**：露出 → 色温度/ティント → コントラスト → 彩度 の順で適用（強度に希釈されずフル効果）
5.5. **色相カーブ**（§5.8）：手順 5 終了時点（base 格子）の色相 h を基準に、Hue vs Hue（色相回転）と Hue vs Sat（彩度ゲイン）の残差を適用する。L* は不変。低彩度域は重み w で減衰し中立色を保護する
6. **残差カーブ加算**（§5.7）：ユーザーがカーブエディタで編集した残差を格子に加算する。R/G/B は格子入力座標 in_c に対し `out_c += r_c(in_c)`、マスターは格子入力のガンマ空間 Rec.709 luma `Y'_in` で評価した 1 つの値を全チャンネルへ同量加算する `out_c += r_master(Y'_in)`（色相不変）。クランプ前に行い、単調性は強制しない
7. **0–1 にクランプ**（最終ステップのみ）

**手動調整の作用空間**：各パラメーターは以下の空間・定義で LUT に焼き込む。

| パラメーター | 作用空間・定義 |
|---|---|
| 露出 | リニア RGB へ変換し `×2^EV` を乗算 |
| 色温度 / ティント | Lab（D65）へ変換し、色温度＝`b*` オフセット、ティント＝`a*` オフセットを加える |
| コントラスト | ガンマ（sRGB）空間・**ピボット 0.5** の S カーブ |
| 彩度 | リニア輝度 `Y = 0.2126R + 0.7152G + 0.0722B`（Rec.709 係数）とのブレンドで増減 |

### 5.6 .cube 出力形式

```
TITLE "<ユーザー指定名>"
LUT_3D_SIZE 33
DOMAIN_MIN 0.0 0.0 0.0
DOMAIN_MAX 1.0 1.0 1.0
<r g b>  # R が最も速く回る順序、小数 6 桁
```

TITLE に埋め込む文字列は互換性のため**サニタイズ**する：ダブルクォート（`"`）と**バックスラッシュ（`\`）**を除去し、印字可能 ASCII 以外（日本語等）は `_` に置換または除去。サニタイズ後に空になる場合は既定名を使用。ダウンロードファイル名自体は日本語可。

**シリアライズ規約**：

- **改行は LF（`\n`）固定**
- 数値は `toFixed(6)`（小数 6 桁固定・**指数表記を排除**）
- 大サイズ（65³ = 274,625 行）のシリアライズは、文字列連結ではなく**配列に push して最後に `join('\n')`** で行い、Worker 側で実行する（メインスレッドをブロックしないため）

### 5.7 カーブエディタ（実効カーブ・残差編集・ヒストグラム）

生成された LUT を、簡素化した DaVinci Resolve 風の RGB カーブ UI で直接編集できる機能。マスター／R／G／B の 4 チャンネルをタブで切り替え、各チャンネルは「実効カーブ（表示）」と「残差編集（入力）」の 2 層で構成する。

**実効カーブ F（記述的）**：カーブエディタに表示するカーブは、残差適用前・クランプ前の base 格子（§5.5 の手順 5 終了時点＝強度ミックス・手動調整の適用後、Float64 で保持）に解析サンプル（§5.1 の有効画素サンプル）を trilinear 適用し、ガンマ（sRGB）空間で入力値ごとの出力期待値 `E[out_c | in_c]` をビン回帰（入力ビン数 `CURVE_BINS`、既定 **64**）して得る。マスターは `E[Y'_out | Y'_in]`（ガンマ空間 Rec.709 luma。表示・適用・ヒストグラムのすべてで同一の `gammaLuma` 定義を用いる）。解析サンプルが 1 件も落ちないビンは、前後の充填ビン間を線形補間し、両端の空ビンは最寄りの充填ビン値を保持する。全ビンが空の場合は恒等 `F(x) = x` とする。

**残差編集 (x, dy)**：ユーザーがカーブエディタ上に置くコントロールポイントは、実効カーブ F からの**出力オフセット（残差）**として保持する。複数点は x 昇順の **Fritsch–Carlson monotone-cubic Hermite** 補間でつなぎ、区間内でのオーバーシュートを起こさない。x=0 と x=1 の端点は常設（dy は編集可・x は固定・削除不可）とし、最外点より外側は端点 dy を定数保持する。1 チャンネルあたりのコントロールポイント上限は `MAX_CONTROL_POINTS`（既定 **16**）、隣接点の最小 x 間隔は `CURVE_MIN_X_GAP`（既定 **0.01**）。

**LUT への適用**：残差は §5.5 の手順 6 として、クランプ直前に格子へ加算する。R/G/B チャンネルは格子入力座標 in_c を残差カーブの入力として `out_c += r_c(in_c)`。マスターは格子入力のガンマ空間 Rec.709 luma `Y'_in` を入力として評価した 1 つの値を**全チャンネルへ同量加算**する（`out_c += r_master(Y'_in)`）。これによりマスターは色相を変えない中立な調整になる。per-channel 適用（`out_c += r_master(in_c)`）は表示と実効果の乖離・マスターであるにもかかわらず色相がずれるという実害があるため不採用とした（§13）。

**表示＝実効果の一致**：R/G/B タブは「表示カーブ（F+Δ）」と「実際の LUT の入出力」が厳密に一致する（残差が入力座標のみの決定的関数であり、条件付き期待値 `E[out|in]` の外に出ないため）。マスタータブは luma 適用により `E[Y'_out | Y'_in]` が厳密に一致する。

**フィードバックループ防止**：実効カーブ F と Source ヒストグラムは、残差適用前の base 格子・解析サンプルから計算する。したがってカーブを編集しても F と Source ヒストグラムは変化しない（編集の基準点が動かない）。結果ヒストグラムのみ残差適用後の最終 LUT を解析サンプルに適用して毎回集計し、カーブ編集にライブで追従する（クリッピング検知が主目的）。

**ゴールデン不変**：カーブが未編集（`isEmptyEdits`：全チャンネルの点が 0 個、またはすべての点で dy=0）のとき、`generateLut` は残差加算のコードパスを一切通らず、既存の最終ループのみを実行する。したがって既存のゴールデンスナップショットとビット一致する。

**単調性・鋭さの制約**：F+Δ（実効カーブ＋残差）の単調性は保証しない。ユーザーが意図的に反転・S 字を作れる創作の自由を優先する（DaVinci Resolve のカスタムカーブと同様）。ただし残差は最終的に LUT の N³ 格子（既定 33）へ焼き込まれるため、残差の実効的な鋭さ（急峻さ）は格子解像度 N に律速される。N を大きく（65 等）取るほど、より鋭いカーブ編集が LUT に忠実に反映される。

**強度との関係**：強度（Identity ミックス）はカーブ編集には**適用されない**。カーブ編集は手動調整（§4.4）と同じく、常にフル効果で LUT に焼き込まれる（§5.5）。

### 5.8 色相カーブ（Hue vs Hue / Hue vs Sat）

RGB カーブ（§5.7）は縦軸が「出力値」であるため、色相を選択的に補正する（肌だけ・空だけ等）ことは原理的にできない。これを埋めるため、色相を横軸に取る 2 本の残差カーブ（Hue vs Hue：色相回転、Hue vs Sat：彩度ゲイン）を追加する。

**色空間**：Lab（D65）の極座標表現 LCh を用いる。既存の `linearRgbToLab` / `labToLinearRgb`（colorspace.ts）を再利用し、極座標変換 `labToLch` / `lchToLab` を colorspace.ts に新設する。色相 h は `atan2(b*, a*)` を [0, 1) へ正規化した値（turns）、クロマ C は `√(a*² + b*²)`。

**横軸の基準（WYSIWYG）**：色相カーブの横軸 h_in は §5.5 の手順 5 終了時点（手動調整までを適用した base 格子。残差カーブ・クランプ前）の色相を用いる。LUT の入力色相（手順 1 の格子座標）を基準にする案は、自動マッチで色相が回転した後にプレビューで見えている色とカーブの軸がずれるため不採用とした（DaVinci Resolve のノード後段カーブと同じ意味論。オーナー決定、§13）。Hue vs Hue と Hue vs Sat は**同じ h_in**で評価し、適用順序に依存しない。

**Hue vs Hue（色相回転）**：横軸=色相 h、縦軸=残差 dy ∈ [−1, +1]。色相回転量は `Δh = dy × 60°`（内部定数 `HUE_CURVE_MAX_ROTATION_DEG`、既定 **60**）。L*・C は不変。

**Hue vs Sat（彩度ゲイン）**：横軸=色相 h、縦軸=残差 dy ∈ [−1, +1]。彩度ゲインは `C' = C × (1 + dy)`（0〜2 倍、下限は C'≥0 でクランプ）。乗算式のため無彩色（C=0）は常に不変。L*・h は不変。

**低彩度減衰（中性色保護）**：クロマが小さい画素は色相自体が数値的に不安定でスペックルの原因になる（HM の残差平滑化・§5.3 と同じ思想）ため、重み `w = smoothstep(0, C0, C)`（`C0 = HUE_CURVE_MIN_CHROMA`、既定 **8.0**、Lab クロマ単位）を Δh・Hue vs Sat の dy の双方へ乗じる。C=0（グレー軸）では w=0 となり両カーブとも完全に無効化されるため、グレーの不変性を数値的に保証する。なお sRGB→XYZ 行列の丸めにより完全なグレーでも Lab 上で ~1e-5 の微小クロマが生じるため、`C ≤ HUE_CURVE_CHROMA_EPS`（既定 **1e-3**）では w を厳密に 0 とするデッドゾーンを設ける（グレー格子点の往復誤差と色相ヒストグラムのノイズスパイクを防ぐ数値ガード。可視域の smoothstep 挙動には影響しない）。

**適用**：base の RGB（ガンマ）→ リニア → Lab → LCh へ変換して h_in・C・L* を得たのち、`h' = h + Δh(h_in)·w`、`C' = C × (1 + dy_sat(h_in)·w)`、L* は据え置きで Lab → リニア → ガンマへ戻す。中間クランプは行わない（範囲外値は符号付きガンマ `sign(x)·γ(|x|)` で扱う既存規約に従う、§5.4）。

**パイプライン上の位置**：§5.5 の手順 5（手動調整）と手順 6（残差カーブ加算）の間に**手順 5.5** として挿入する。RGB カーブの実効カーブ F（§5.7）は手順 5.5 適用**後**の格子から計算するため、R/G/B/マスタータブの「表示＝実効果の一致」は色相カーブ併用時も維持される。一方、色相カーブ自身の横軸ヒストグラム（後述）は手順 5.5 適用**前**（base）から計算し、自軸が自分の編集に依存しないフィードバックループ防止構造（§5.7 と同じ）を踏襲する。

**周期スプライン**：色相軸は 0 と 1（0°と 360°）が繋がる周期軸である。core/curve.ts に、Fritsch–Carlson monotone-cubic Hermite 補間を円環上で評価する周期版の関数を新設し、区間の継ぎ目（0/1 境界を含む）で C1 連続を保証する。§5.7 の RGB カーブと異なり x=0 / x=1 の端点は常設しない（点 0 個で恒等 dy=0、点 1 個で全周一定の dy）。最小 x 間隔 `CURVE_MIN_X_GAP`（§5.7 と共通の定数、既定 0.01）は円環距離（circular distance）で判定する。コントロールポイント数上限は `MAX_CONTROL_POINTS`（§5.7 と共通、既定 16）を Hue vs Hue・Hue vs Sat それぞれに独立して適用する。

**データモデル**：`CurveEdits` に `hue?: ControlPoint[]` / `hueSat?: ControlPoint[]` をオプショナルフィールドとして追加する。`isEmptyEdits`（§5.7）は両フィールドについても、未指定・点が 0 個・全点で dy=0 のいずれかであれば空編集とみなすよう拡張する。

**ヒストグラム**：hist 転送バッファの既存 4 ブロック（R / G / B / Y'、§5.7）に、色相用の第 5 ブロック H（`HIST_BINS`、周期軸 [0, 1)）を追加する。Source 側は base 格子適用後の解析サンプルの色相を、Result 側は最終 LUT 適用後の解析サンプルの色相を、いずれも低彩度減衰と同じ重み w を掛けて集計する（グレー画素の数値的に無意味な色相がヒストグラムを占有するのを防ぐ）。

**ゴールデン不変**：`hue` / `hueSat` が未編集（拡張後の `isEmptyEdits` が真）のとき、`generateLut` は手順 5.5 のコードパスを一切通らず、既存の最終ループのみを実行する。したがって未編集時は既存のゴールデンスナップショットとビット一致する。

**強度との関係**：§5.7 の RGB カーブ・手動調整と同じく、強度（Identity ミックス）は色相カーブに**適用されない**（常にフル効果）。

**格子解像度の注意**：色相・彩度の残差も最終的に LUT の N³ 格子（既定 33）へ焼き込まれるため、鋭い色相選択（狭い色相帯だけを補正する等）は格子解像度 N に律速される（§5.7 と同じ注意）。

## 6. UI 仕様

### 6.0 設計原則

1. **ウィザードにはしない**：画面分割はせず全機能を 1 画面に置く。調整⇄比較を行き来する使い方を妨げないため。操作の誘導はドロップゾーンのガイド文言・点滅ハイライトで行う（進行インジケーターは置かない）
2. **プレビュー常時可視**：パラメーター操作中、結果が必ず画面内に見えていること（本アプリ最重要の UX 要件）
3. **LUT 生成が主・手法は並列**：本アプリの主目的は LUT の生成であり、参考画像からの自動マッチは手法の一つと位置づける。手法は 3 つのアコーディオン——「**色合わせ**」（Reference ドロップゾーン・用途説明ヒント・モード 3 択・強度・ノイズ抑制。見出しは §4.2）、「**カーブ**」（§5.7・§5.8）、「**詳細調整**」（露出以下 7 項目、スムージングを含む）——として対等に並べ、いずれも既定：閉。常時表示のコントロール群は置かず、常時可視なのはプレビューと書き出しバーのみ。アコーディオンの開閉はすべてユーザーの手動操作とし、自動制御は行わない（§6.2）
4. **書き出し導線は固定**：書き出しバーはデスクトップ・モバイルとも画面下部 sticky の全幅バー（常に画面内）

### 6.1 ブレークポイントとレイアウト

| 区分 | 幅 | レイアウト |
|---|---|---|
| モバイル | < 768px | 1 カラム縦積み＋2 つの sticky（下記） |
| タブレット | 768–1119px | 2 カラム（左レール 240px ＋ プレビュー） |
| デスクトップ | ≥ 1120px | 2 カラム（左レール `clamp(300px, 28vw, 420px)` の流動幅 ＋ プレビュー）、余白拡大 |

**デスクトップ / タブレット（2 カラム）**

- 左レール（上から）：「色合わせ」アコーディオン（Reference ドロップゾーン・サンプル参考画像リンク・用途説明ヒント・モード 3 択〔セグメントコントロール〕・強度スライダー・ノイズ抑制スライダーを格納） → 「カーブ」アコーディオン（§5.7） → 「詳細調整」アコーディオン → すべてリセット。レール自体が縦スクロール
- 右カラム：プレビュータブ（元画像 / 適用後 / 比較〔既定〕。タブバー右端に Source の「差し替え」「削除（×）」ボタンを配置）→ 比較スライダー付きプレビュー（`position: sticky` でスクロールに追従）。**空状態ではプレビュー全面が Source のドロップゾーン**となり、ガイド文言（ドラッグ＆ドロップ / クリックで選択・対応形式）と「サンプル画像で試す」リンクを表示する（§6.2）
- 書き出しバー（サイズ選択 / ファイル名 / DL ボタン）はグリッドの外、画面下部に全幅 sticky で常時表示（モバイルと共通、下記）
- **Reference 投入時のみ**、参考画像のサムネイルをプレビュー右上に小窓表示（タップで一時拡大）し、「目標」を常に参照できるようにする（Reference 未投入時は小窓自体を表示しない）

**モバイル（1 カラム）**

- 上から：ヘッダー（ロゴ / JA|EN / ?）→ **プレビュー（画面上部に sticky。空状態は Source ドロップゾーン＋「サンプル画像で試す」リンク）** → 「色合わせ」アコーディオン → 「カーブ」アコーディオン → 詳細調整アコーディオン → すべてリセット → **書き出しバー（画面下部に sticky）**
- プレビューを上部 sticky にすることで、下のスライダーをスクロール操作している間も結果が見え続ける（原則 2 の実現手段）
- sticky プレビューの高さは画面高の最大 40% にクランプし、スライダー操作領域を確保する。高さ計算には `dvh` 単位または `visualViewport` API を用いる（iOS Safari のアドレスバー可変により `vh` がずれる問題への対策）。空状態のドロップゾーンのガイド文言・サンプルリンクはこの高さ内で視認できるよう構成する

### 6.2 状態と遷移

1. **初期（空状態）**：プレビュー全面の Source ドロップゾーン＋「サンプル画像で試す」リンク、および「色合わせ」アコーディオン内の Reference ドロップゾーン＋サンプル参考画像リンクが有効。調整・書き出しはグレーアウト。点滅等の誘導は行わない（静穏な初期状態）。**サンプルは入力口ごとに分割**する：プレビューのリンクはサンプル Source のみ、「色合わせ」内のリンクはサンプル Reference のみを読み込む（同梱デモペア public/ の対応する側）。これにより「サンプル元画像＋手動 LUT 作成」「自分の元画像＋サンプル参考画像でマッチ」の組み合わせも可能になる
2. **Source のみ投入**：**正規の作業状態**（Reference なしの手動 LUT 作成モード、§4.1/§4.3）。手動調整パネル・カーブエディタが有効になり、恒等変換を基底に編集できる。モード 3 択・強度・スムージング・ノイズ抑制などの自動マッチ系コントロールは無効（理由をツールチップ表示、§4.4）。追加の誘導は行わず、「色合わせ」アコーディオン内の Reference ドロップゾーンのガイド文言「参考画像を入れると自動で色を合わせます（任意）」が案内を担う
3. **Reference のみ投入（Source 未投入）**：プレビュー空状態の Source ドロップゾーン（ガイド文言）を点滅等で誘導する。**点滅誘導はこのケースに限定**する
4. **両画像投入**：自動マッチ経路（§4.2/§4.3）へ移行し、全機能が有効になる
5. **計算中**：プレビュー領域にスケルトン＋プログレス（Worker からの進捗イベント）。UI 操作は可能
6. **生成済み**：全機能有効（有効条件は §4.4 に従う）。パラメーター変更は即時反映、モード変更・画像差し替えは再計算。カーブ編集（残差）はスライダー値と同じ扱いで、モード変更・スライダー変更・画像差し替えをまたいで保持される。破棄は 3 段階のスコープで行う：①「カーブ」パネル内の「◯◯をリセット」（選択中タブのチャンネルのみ。ラベルはタブに応じ動的に変わり、そのチャンネルが未編集なら disabled）または「すべてのカーブをリセット」（カーブ全体。従の位置づけの小型ボタンで、全チャンネル未編集なら disabled）、②「詳細調整」アコーディオン内の「詳細調整をリセット」（スムージング/露出/コントラスト/彩度/色温度/ティント/ブラック保護の詳細 7 項目のみ。強度・ノイズ抑制・カーブは対象外）、③左レール末尾の「すべてリセット」（強度・ノイズ抑制・詳細調整・カーブ編集をまとめて初期化。自動マッチ結果は保持）。**Reference を後から投入すると手動 LUT 作成モードから自動マッチ経路へ移行し、手動調整・カーブ編集はそのまま保持される。Reference を削除すると手動 LUT 作成モードへ戻る（編集は保持される）**
7. **エラー**：非対応形式・デコード失敗・メモリ不足はトースト表示で通知し、直前の状態を維持する（`>50MP` の事前サイズ検出は廃止し、`createImageBitmap` の失敗を捕捉する方式に統一。§4.1 参照）。HEIC 等の非対応形式では「JPEG に変換してください」等の案内を文言に含める
8. **アコーディオンの自動開閉は行わない**：開閉はすべてユーザーの手動操作とする。旧仕様の「Reference 投入/削除への開閉連動」「サンプル実行時の自動オープン」はいずれも廃止（Reference とそのサンプルの入口がアコーディオン内へ移り、操作時点でアコーディオンは必然的に開いているため。§13）

### 6.3 インタラクション詳細

- **比較スライダー**：デスクトップは境界線上どこでもドラッグ可＋ハンドル。**モバイルはハンドル（直径 44px 以上）のドラッグのみで動作**させ、縦スクロールとのジェスチャー競合を回避する。キーボード（←→）対応
- **スライダー全般**：タッチターゲット 44px 以上。数値ラベルをタップすると直接数値入力できる。ダブルタップ / ダブルクリックで個別リセット
- **カーブエディタ**（§5.7・§5.8）：タブ [マスター][R][G][B][色相][彩度] で編集チャンネルを切り替える。ハンドルのヒット半径は 24px（直径 48px、44px 以上を満たす）。点の追加は曲線近傍をタップ／クリックした瞬間に即時追加する（300ms 待ちなし。ヒットテストが空間的に排他なため誤操作しない）。削除はハンドル上のダブルタップ／ダブルクリック限定（RGB 系タブの端点は削除不可、色相／彩度タブは端点自体が存在しない）。ドラッグ中はプレビューを draft 品質で更新し、ドラッグ確定後にフル品質へ戻す（他のスライダーのドラッグと同じ扱い）。canvas は `touch-action: none` を指定し、縦スクロールとのジェスチャー競合を避ける（この指定はプレビュー canvas とカーブ canvas の両方に適用する）。背景は選択チャンネルの Source ヒストグラム（半透明塗り）＋結果ヒストグラム（輪郭線、クリッピング検知用）。Source ヒストグラムが疎な範囲では基底カーブ（実効カーブ F）を減光表示し、推定であることを示す（マスター／R／G／B タブのみ。色相／彩度タブに実効カーブ F の概念はない）
- **色相／彩度タブ**（§5.8）：横軸は色相の周期軸（0–360°）のため x=0/x=1 の固定端点はなく、ドラッグで一周ラップして操作できる。背景に x 軸に沿った色相グラデーション帯（LCh を固定 L*・C* で sRGB へ変換して描画）と色相ヒストグラムを表示する。中央の水平線が恒等（dy=0）、縦軸は dy ∈ [−1, +1]。タップ追加・ドラッグ・ダブルタップ削除等の操作系は他タブと共通
- **プレビュー**：拡大縮小・パン機能は持たない。画像は常にステージへフィット表示する
- **書き出しバー**：DL ボタンは常時 1 つのプライマリボタン。ファイル名・サイズ選択は幅 1120px 未満ではバー内の展開トグルで表示し、既定値ならワンタップで保存完了。幅 1120px 以上は全項目をバー内に横 1 行展開する

### 6.4 補助 UI・ビジュアル

- ダークテーマ基調（色評価を妨げないニュートラルグレー背景）。UI のアクセント色は彩度低めに抑え、プレビューの色を邪魔しない
- 各スライダーにツールチップで一言説明（i18n 対象）
- `?` ボタンで使い方モーダル（3 ステップ図解＋「LUT が扱える範囲」の注記）
- 「参考画像の色の雰囲気を移す機能。被写体・明るさが近い 2 枚ほど自然に仕上がり、まったく異なる場面同士では合わない」旨の用途説明ヒントを「色合わせ」アコーディオン内・Reference ドロップゾーンの直下に常設表示する（自動マッチという一手法に固有の注意書きのため、画面全体での常時表示は廃止。§13）

| 項目 | 目標 |
|---|---|
| 自動マッチ計算時間 | 1 秒以内（512px 解析、一般的なノート PC） |

**サンプル画像**：`public/` に同梱するデモ用サンプルペアは、**オーナー自身が権利を持つ実写写真**を用いる（第三者の写真・権利未確認の外部素材は使わない）。実写はマッチ品質のデモとして合成画像より説得力があるため。

**Worker とデータ受け渡し（§3 補足）**：

- メイン ⇄ Worker 間のピクセル・LUT データの受け渡しは、**Transferable な `ArrayBuffer`** を用いる（`postMessage` の transfer 指定でコピーを避ける）。`SharedArrayBuffer` は使わない（COOP/COEP ヘッダ設定が不要になり、Cloudflare Pages の静的ホスティングと相性が良いため）
- **プレビュー描画はメインスレッドで行う**（WebGL / Canvas 2D はメインスレッド）。Worker は統計計算・LUT 生成・.cube シリアライズ等の重い純粋計算に限定する

## 11. テスト方針

- **ユニット**：色空間変換の往復誤差、MKL の閉形式解（既知の共分散で検証）、HM の単調性、.cube のフォーマット妥当性
- **ノイズ抑制（§5.3）**：①s=0 で現行実装（σ=2・クランプなし）と完全一致 ②クランプ＋残差平滑化後もカーブが単調 ③CLAHE 式再分配で端点（総上昇量）が保存される ④処理後カーブの最大傾き ≤ S_max(s) ⑤s=100 でカーブが恒等＋定数オフセットに近い ⑥Source＝Reference の同一性がスライダー値によらず維持される
- **カーブエディタ（§5.7）**：①残差カーブの評価・格子加算（未編集時は既存ゴールデンとビット一致すること）②実効カーブ回帰・ヒストグラム（空ビンの線形補間・全ビン空での恒等 F=x）③実効カーブがカーブ編集（残差）に非依存であること（フィードバックループ防止の検証）④マスター残差のガンマ空間 luma 適用が全チャンネルへ同量加算され色相を変えない（色相不変）こと
- **色相カーブ（§5.8）**：①周期補間の継ぎ目（0/1 境界）で C1 連続であること ②hue／hueSat が未編集のとき手順 5.5 のコードパスを通らず既存ゴールデンとビット一致すること ③クロマ C=0（グレー画素）では Δh・彩度ゲインとも重み w=0 で完全に無効化されること（グレー不変）④既知の Lab 色（例：既知の a*/b* から求まる h・C）に対する Hue vs Hue の回転量・Hue vs Sat の彩度ゲインが理論値と一致すること
- **ゴールデンテスト**：固定入力ペアに対する LUT 出力のスナップショット比較（アルゴリズム回帰検知）
- **同一性テスト**：Source＝Reference のとき LUT ≒ Identity になること。**スムージング 0・強度 100%** の条件で実施し、全格子点の最大誤差が許容値（例 `1e-3`）以下であることを検証する
- **実写検証**：デジタル写真×フィルム写真のペア数組で目視評価。加えて **iPhone の HDR（ゲインマップ付き）写真**のペアを検証対象に含め、広色域・HDR 由来の入力でも破綻しないことを確認する
- **外部互換**：書き出した .cube を DaVinci Resolve / Photoshop で読み込み、ブラウザプレビューと一致することを確認

# 仕様書：参考画像ベース LUT ジェネレーター（仮称：LUT Match）

- 版数: v0.2
- 最終更新: 2026-07-09
- ステータス: 確定

---

## 1. 概要

### 1.1 目的

「元画像（Source）」と「参考画像（Reference）」の2枚を入力すると、元画像の色を参考画像の雰囲気に近づける 3D LUT を自動生成し、`.cube` ファイルとして書き出せる Web アプリを開発する。

例：デジタルカメラで普通に撮った写真（Source）と、昔フィルムカメラで撮った写真（Reference）を入力すると、フィルム風の色合いになる LUT が得られる。

### 1.2 コンセプト

- **完全ローカル処理**：画像はサーバーに一切送信しない。すべてブラウザ内（JavaScript / Web Worker）で計算する
- **自動＋手動**：自動カラーマッチで 8 割を作り、手動パラメーターで追い込む
- **汎用出力**：生成した LUT は DaVinci Resolve / Premiere Pro / Photoshop / Lightroom（プロファイル経由）/ OBS など `.cube` 対応ソフトでそのまま使える

### 1.3 スコープ外（v1 では実装しない）

- 動画ファイルの入力
- グレイン・ビネット等の空間エフェクト（LUT は色の写像のみで原理的に含められないため）
- サーバーサイド処理、アカウント機能、課金
- プリセット保存、PNG HALD 出力（将来検討 → §12）

## 2. 用語

| 用語 | 意味 |
|---|---|
| Source | LUT を適用したい元画像 |
| Reference | 目標とする雰囲気の参考画像 |
| 3D LUT | RGB 入力色 → 出力色の 3 次元ルックアップテーブル |
| `.cube` | Adobe/IWLTBAP 系で標準的な LUT テキスト形式 |
| MKL | Monge-Kantorovich 線形化。線形変換による色分布マッチ |
| HM | ヒストグラムマッチング。チャンネル毎の累積分布を一致させる非線形マッチ |
| Identity LUT | 入力＝出力の無変換 LUT。強度ミックスの基準に使う |

## 4. 機能要件

### 4.1 画像入力

- Source / Reference の 2 つのドロップゾーン。ドラッグ＆ドロップとファイル選択の両対応
- 対応形式：JPEG / PNG / WebP（ブラウザがデコードできるもの）
- **デコード経路**：`createImageBitmap(file, { imageOrientation: 'from-image', resizeWidth, resizeHeight })` を用いて、目的の解像度へ縮小したうえで `ImageBitmap` を得る。**フル解像度を一度も canvas に描画しない**（iOS Safari の canvas 面積上限〔約 16.7MP〕を超えて描画が失敗するのを避けるため）。`imageOrientation: 'from-image'` により EXIF 回転はデコード時に適用される
- 入力画像は解析用に長辺 **512px** へ縮小して統計計算に使用（プレビュー表示用は長辺 2048px まで別途保持）。サムネイル表示も同一のデコード経路（`createImageBitmap` + `resizeWidth/resizeHeight`）を使い回し、回転ズレやオリエンテーション不一致を防ぐ
  - 根拠：MKL（平均・共分散）は数千 px、HM（256 ビン CDF）は長辺 256px 相当で概ね足りるが、シャドウ・ハイライトの分布の裾の推定安定性のため余裕を持たせる。処理時間の支配項は元画像のデコード/リサイズであり、512px の統計計算コストは数十 ms と無視できる。内部定数 `ANALYSIS_LONG_EDGE` として変更可能にする
- 透過 PNG はアルファ 50% 未満の画素を統計計算から除外する
- 広色域画像（Display P3 等）は Canvas 経由で sRGB に変換して扱う（制限としてヘルプに明記）。canvas / `ImageData` の `colorSpace` は既定（`'srgb'`）を明示指定して扱う
- **巨大画像の扱い**：`>50MP` を事前検出してエラーにする方式は廃止する（`createImageBitmap` でフル解像度を描画しないため事前サイズ判定の必要がなくなる）。代わりにデコード失敗・メモリ不足（`createImageBitmap` の reject 等）を捕捉し、トーストで通知して直前の状態を維持する
- **HEIC 等の非対応形式**：ブラウザがデコードできない形式はデコード失敗として捕捉し、エラー文言に「JPEG に変換してください」等の具体的な案内を含める
- 画像差し替え時は直前の `ImageBitmap` を `ImageBitmap.close()`、生成済み Object URL を `URL.revokeObjectURL()` で明示解放してから再計算する

### 4.2 自動カラーマッチ（3 モード）

| モード | UI 表示名（日/英） | 内部処理 | 想定用途 |
|---|---|---|---|
| A | ナチュラル / Natural | MKL のみ | 破綻を避けたい・被写体が大きく異なる場合 |
| B | 忠実 / Faithful | HM のみ（チャンネル別） | トーンカーブごと強く寄せたい場合 |
| C（既定） | バランス / Balanced | HM → MKL → HM の複合パイプライン | 通常はこれ。非線形の再現と安定性の両立 |

- モード変更時は自動で再計算（Web Worker、目標 1 秒以内）
- 計算は知覚均等性のため **リニア化した RGB** 上で行う（sRGB EOTF でデガンマ → 計算 → 再ガンマ）。Lab 空間案との比較は検討の結果不採用で確定（§13。HM のみガンマ空間で動く意図的な例外 → §5.3）

### 4.3 LUT 生成

1. §4.2 の変換関数 f(r,g,b) を得る（モード B/C では HM カーブ構築に**ノイズ抑制**スライダー連動の傾きクランプ・残差平滑化パラメータ化が入る。詳細は §5.3）
2. N×N×N（既定 33）の格子点すべてに f を適用して 3D LUT 化
3. **範囲外挙動**：入力画像に存在しない色域（統計の外挿域）は Identity 方向へ減衰させ、極端な色化けを防ぐ
4. **平滑化**：3D ガウシアンぼかし（**スムージングスライダー連動**）でバンディング・色飛びを抑制

処理の詳細な順序は §5.5 に従う（外挿減衰 → 平滑化 → 強度ミックス → 手動調整 → クランプ）。

### 4.4 手動調整パラメーター

生成された LUT に対して、以下を LUT に「焼き込む」形で追加調整できる。すべてリアルタイムでプレビューに反映。

| パラメーター | 範囲 | 既定値 | 備考 |
|---|---|---|---|
| 強度（ミックス） | 0–100% | 85% | **自動マッチ結果と Identity のブレンド率（手動調整には影響しない）** |
| スムージング | 0–100 | 20 | §4.3 の平滑化強度 |
| ノイズ抑制 | 0–100 | 15 | HM カーブの傾き急変を抑制し、ノイズの増幅を防ぐ（ノイズ除去そのものではない）。モード A（ナチュラル）では無効。§5.3 |
| 露出 | −2.0〜+2.0 EV | 0 | |
| コントラスト | −50〜+50 | 0 | 中間グレー基準の S カーブ |
| 彩度 | −100〜+100 | 0 | |
| 色温度 | −100〜+100 | 0 | Lab の b* 軸相当 |
| ティント | −100〜+100 | 0 | Lab の a* 軸相当 |
| ブラック保護 | 0–20% | 5% | 指定した**知覚（sRGB）輝度**以下を統計マッチから除外し、締まった黒を維持。内部では sRGB→リニア変換した値でリニア輝度と比較する |

- 「リセット」ボタンで手動調整のみ初期化（自動マッチ結果は保持）

### 4.5 プレビュー

- Source に現在の LUT（自動＋手動込み）を適用した結果を常時表示
- **ビフォーアフター比較スライダー**：1 枚の画像の上で境界線をドラッグして適用前/後を左右比較（初期位置 50%）
- 表示切替タブ：`元画像` / `適用後` / `比較スライダー`（既定）
- Reference はサムネイルとして常時参照表示
- 画像は常にステージへフィット表示する（拡大縮小・パン操作はなし）
- **描画経路**：WebGL2 の 3D テクスチャに LUT を格納する経路を主経路とする（2D パック LUT 案は採用しない）。`webglcontextlost` / `webglcontextrestored` をハンドリングし、コンテキスト喪失時は復旧を試み、復旧に失敗した場合は Canvas 2D フォールバックへ自動切替する（詳細は §7）
- **バッキングストア解像度**：`devicePixelRatio` を上限 2 でクランプして決定する（高 DPI 端末での過大な描画コスト・メモリ消費を避ける）

### 4.6 書き出し

- 形式：`.cube`（3D LUT）
- サイズ選択：**17 / 33 / 65**（既定 33）
- 出力内容＝自動マッチ＋手動調整をすべて合成した最終 LUT
- ファイル名：ユーザー入力（既定 `lutmatch_YYYYMMDD_HHmm.cube`）。名前は `TITLE` ヘッダーにも埋め込む
- 併せて「適用後画像を PNG 保存」ボタンも用意（確認用）。保存はプレビュー解像度（長辺 **2048px**）で行う

### 4.7 UI 言語（日英切替）

- 日本語 / English の 2 言語。ヘッダーのトグルで即時切替
- 初期言語はブラウザの `navigator.language` から自動判定（`ja*` → 日本語、それ以外 → 英語）
- 選択は `localStorage` に保存
- 言語切替時は `<html lang>` 属性を選択言語（`ja` / `en`）に同期する
- 文言は `src/i18n/ja.ts` / `en.ts` の辞書オブジェクトで一元管理

## 5. アルゴリズム仕様

### 5.1 前処理

1. 画像を長辺 512px に縮小、ピクセル配列を取得
2. sRGB → リニア RGB へ変換
3. ブラック保護しきい値未満の画素を統計対象から除外（UI の知覚 % を sRGB→リニア変換した値を Rec.709 **リニア輝度**と比較。Source / Reference 両方に対称に適用する）
4. **有効画素数の保護**：アルファ除外（§4.1）とブラック保護を適用した後の有効画素数が `N_MIN_PIXELS`（既定 **1024**）未満の場合、統計推定が不安定になるため §5.2 と同様に**平均シフトのみのフォールバック**へ切り替え、UI に警告を表示する

### 5.2 MKL（モード A）

Source / Reference それぞれの平均ベクトル μ と共分散行列 Σ を計算し、閉形式解

T = Σs^(-1/2) · (Σs^(1/2) · Σr · Σs^(1/2))^(1/2) · Σs^(-1/2)

により線形写像 `f(x) = T(x − μs) + μr` を得る（Pitié & Kokaram 2007）。3×3 の固有値分解のみで済むため軽量。

**退化・破綻対策**：実画像の**異方的な色分布**では `Σs^(-1/2)` の小固有値方向のゲインが爆発し、MKL は 3×3 でチャンネルを混ぜるため微小な粒状ノイズを彩度・色相ノイズ（マゼンタ/緑のスペックル、青・赤の過強調）へ増幅する。これを次の 3 段で抑える。

- **収縮正則化（絶対下限つき）**：共分散を正則化する `Σ ← Σ + εI`、`ε = max(trace(Σ)/3 × 1e-4, MKL_EPS_ABS)`。相対量 `trace/3 × 1e-4` だけでは暗部（低分散）画像で `ε≒0` となり無力なため、8bit 量子化ノイズ相当のリニア分散を**絶対下限** `MKL_EPS_ABS`（既定 **1e-5**、σ≒0.0032 リニア）として課す。Source・Reference の両共分散へ同一に適用し、行列平方根・逆行列を数値的に安定させる。固有値は `sqrt` を取る前に **0 でクランプ**する
- **ゲイン上限クランプ**：輸送行列 T は対称正定値。構築後に固有値分解し、方向別ゲイン（固有値）を `[1/MKL_MAX_GAIN, MKL_MAX_GAIN]`（既定 `MKL_MAX_GAIN = **4.0**`）にクランプして再構成する。上限 4.0 はリニア空間で 2 EV（4×）に相当し、露出・コントラスト差の**正当な全域スケーリングは素通しつつ**、それを超える病的な方向ゲインだけを頭打ちにする。T が数値誤差で厳密対称でない場合は `(T+Tᵀ)/2` で対称化してから分解する。Source=Reference のとき T=I なので同一性は保たれる
- **段階的フォールバック（異方性ブレンド）**：Source 共分散の異方性比 `r = λmin/λmax` に応じ、フル MKL 行列 T と**チャンネル別対角スケール** `D = diag(σr_i/σs_i)`（同じくゲインクランプ・チャンネルを混ぜないため色相ノイズを増幅しない）の間を重み `w` で補間する（`T' = w·T + (1−w)·D`）。`w` は `r ≥ MKL_ANISO_FULL`（既定 **1e-2**）で 1、`r ≤ MKL_ANISO_DIAG`（既定 **1e-4**）で 0、その間は log 空間 smoothstep。これによりニュートラルな Reference やグレースケール Source でも**平均シフトへ即崩壊せず**対角スケールで穏当にマッチする（旧・二値ランク判定 `MKL_RANK_RATIO` は廃止）
- **完全退化のフォールバック**：有効画素数が `N_MIN_PIXELS`（既定 **1024**）未満の場合のみ、線形写像を求めず**平均シフトへフォールバック**する：`T = I`、`f(x) = x − μs + μr`。この場合は UI に警告を表示する

### 5.3 HM（モード B）

R/G/B 各チャンネルについて Source の累積分布関数を Reference の累積分布関数へ写像（256 bin、単調性を保証）。得られる 3 本の 1D カーブを 3D 格子へ適用する。

**ビン割りはガンマ（sRGB）空間で等間隔に定義する。** リニア空間で等間隔ビンを切るとシャドウ側の階調が極端に粗くなり、フィルムルックで重要な暗部の再現が破綻するため。リニア値はビン参照時にガンマ空間へ写像して扱う。

**HM はガンマ（sRGB）空間で動く 1D 単調リマップである。** すなわち HM は「統計計算はリニア RGB 上で行う」原則（§4.2）の**意図的な例外**であり、CDF の構築とカーブ適用はガンマ空間で行う（この方が暗部階調が保たれるため）。この例外は仕様上の確定事項として扱う。

**逆 CDF の求め方**：Reference の逆 CDF は分位点の**線形補間**で求める。得られる写像カーブは**単調非減少であることを保証**する（数値誤差で非単調が生じた場合は単調になるよう補正する）。

**残差平滑化（暗部スペックル対策）**：暗部（リニア輝度 5〜15% 程度）ではソースのヒストグラムが疎になりやすく、逆 CDF 由来のリマップカーブが**単一ビンで急峻に跳ねる傾きスパイク**を作る。このスパイクはソースの微小な粒状ノイズ（±数レベル）を、チャンネル間で不整合な方向の大きな色差へ増幅し、暗部にマゼンタ/緑のスペックルを生む（実機検証で確認）。対策として、カーブから恒等 `y=x` を引いた**残差**にごく弱い 1D ガウシアン（σ=2 ビン、内部定数 `HM_RESIDUAL_SMOOTH_SIGMA`）を掛け、恒等を足し戻してスパイクを近傍ビンへ均す。恒等入力（残差=0）は平滑化後も厳密に恒等のままで**同一性（Source=Reference → LUT≒Identity）を壊さない**。平滑化後に単調化を再適用して単調非減少を保証する。σ を上げるほどスペックルは減るが Reference 分布への追従がわずかに甘くなるトレードオフがあり、σ=2 は暗部の傾きスパイクを 1/5 程度・ギザつき（2 階差分）を 1/19 程度に抑えつつ CDF マッチの平均挙動をほぼ保つ控えめな値。

**ノイズ抑制スライダー（0–100、既定 15）**：3D LUT は画素単位の色変換であり、空間的なノイズ除去は原理的に不可能である。本機能は「変換の勾配（傾き）を抑えてノイズを増幅しない」アプローチであり、ノイズ除去そのものではない。上記の残差平滑化は固定 σ=2 だったが、色境界での色ノイズ（スペックル）が目立つ画像向けに、ユーザーが抑制強度を調整できるようにする。1 本のスライダー値 s は、次の 2 つの機構を**同時に**駆動する（両者は相補的：ガウシアン平滑化は幅の狭い傾きスパイクに、傾き上限クランプは幅の広い急峻区間に効く）。

1. **残差平滑化 σ のパラメータ化**：`σ(s) = HM_RESIDUAL_SMOOTH_SIGMA × 2^(s/25)`（s=0 で σ=2 ビン＝現行の固定値、s=100 で σ=32 ビン）
2. **カーブ傾き上限クランプ**：`S_max(s) = 2^(4(1−s/100))`（s=0 で 16 ≒ 実質無効、s=100 で 1＝恒等＋オフセットへ漸近）。**s=0 ではクランプ処理自体を完全スキップ**し、現行挙動（σ=2・クランプなし）と完全に一致させる

上記の係数マッピングは出発点であり、実装時に実画像ペアで校正する。

**傾きクランプの実装（CLAHE 式再分配）**：HM カーブの微分（ビンごとの傾き `d[i] ≥ 0`）を上限 `S_max` でクリップし、超過分を全ビンへ均等再分配する（CLAHE のクリップリミット再分配と同じ構造。2〜3 回の反復で収束）。`d ≥ 0` を保ったまま操作するため単調性は自動的に保たれ、端点（カーブの総上昇量）も保存される。

**カーブ構築の処理順**：CDF 写像 → 単調化 → 傾きクランプ（CLAHE 式再分配）→ 残差平滑化 → 単調化再保証。クランプを先に行うことで生じる折れ点を後段の残差平滑化が丸め、かつ正規化ガウシアンは微分の最大値を増やさないため、平滑化後も傾き上限の保証が維持される。

**作用範囲**：モード B は HM 1 段、モード C は HM→MKL→HM の 1 段目・2 段目双方に同一パラメータを適用する。モード A（MKL のみ）は HM を使わないため本スライダーは無効（UI はグレーアウトし理由をツールチップ表示）。強度（Identity ミックス）・スムージング（§5.5 の 3D 格子平滑化）・手動調整とは独立したパラメータである。

**実装インターフェース**：`GenerateLutOptions` に `noiseSuppression: number`（0–100）を追加し、UI → Worker → `buildMatchTransform` → `buildHistMatch`（引数追加）の経路で伝搬する設計とする。

### 5.4 複合（モード C・既定）

`HM → MKL → HM` の直列適用（color-matcher の HM-MVGD-HM 構成に準拠、線形段は MKL を使用）。各段は §5.2 / §5.3 の実装を再利用する。

**段間データフロー**：解析画素集合（§5.1 の有効画素サンプル）を各段の変換で順に押し出し、**次段の統計（CDF・平均・共分散）は変換後のサンプルから推定する**。具体的には：

1. **1 段目 HM**：元の Source 分布 → 元の Reference 分布へマッチ。Source サンプルをこのカーブで変換し、中間サンプル S₁ を得る
2. **MKL 段**：S₁ の平均・共分散を Reference の平均・共分散へ合わせる線形写像を求め、S₁ を変換して S₂ を得る
3. **2 段目 HM**：**「HM→MKL 適用後の Source 分布」（= S₂ の分布）を「元の Reference 分布」へマッチ**させる。すなわち S₂ の CDF を元 Reference の CDF へ写像する

**中間段では 0–1 クランプを行わない**（範囲外値を保持したまま次段へ渡す）。段ごとにクランプすると情報が失われ、複合の利点が損なわれるため。クランプは §5.5 の最終ステップのみ。

**範囲外値（負値・>1）の扱い**：中間クランプなしで生じる範囲外値を、段間の色空間変換で破綻させないため次の規約を用いる：

- **ガンマ ⇄ リニア変換**：符号付き拡張 `sign(x)·γ(|x|)` を用いる（負値でも符号を保ったまま変換）
- **HM カーブ**：ビン範囲 `[0,1]` の外側では、端点の傾きで**線形外挿**する

### 5.5 LUT 化と後段

処理順序は以下のとおり（**強度ミックスは自動マッチ結果のみに作用し、手動調整はそのフル効果が保たれる**点に注意）：

1. **N³ 格子点に f を適用**
2. **外挿域の Identity 減衰**：Source 色分布の平均・共分散に基づく**マハラノビス距離**がしきい値 d0 を超える格子点は、距離に応じて f の出力を Identity 方向へ線形に減衰させる（d0 と減衰幅は実装時にチューニング、既定 d0=3.0）。マハラノビス距離は、**格子点座標をリニア化したうえで** Source の**リニア空間統計**（μs, Σs）に対して測る（`d² = (x_lin − μs)ᵀ Σs⁻¹ (x_lin − μs)`）
3. **3D ガウシアン平滑化**（スムージングスライダー連動）
4. **Identity ミックス（強度）**：ここまでの自動マッチ結果を Identity とブレンドする。強度はこの段のみに作用し、以降の手動調整には影響しない
5. **手動調整**：露出 → 色温度/ティント → コントラスト → 彩度 の順で適用（強度に希釈されずフル効果）
6. **0–1 にクランプ**（最終ステップのみ）

**手動調整の作用空間**：各パラメーターは以下の空間・定義で LUT に焼き込む。

| パラメーター | 作用空間・定義 |
|---|---|
| 露出 | リニア RGB へ変換し `×2^EV` を乗算 |
| 色温度 / ティント | Lab（D65）へ変換し、色温度＝`b*` オフセット、ティント＝`a*` オフセットを加える |
| コントラスト | ガンマ（sRGB）空間・**ピボット 0.5** の S カーブ |
| 彩度 | リニア輝度 `Y = 0.2126R + 0.7152G + 0.0722B`（Rec.709 係数）とのブレンドで増減 |

### 5.6 .cube 出力形式

```
TITLE "<ユーザー指定名>"
LUT_3D_SIZE 33
DOMAIN_MIN 0.0 0.0 0.0
DOMAIN_MAX 1.0 1.0 1.0
<r g b>  # R が最も速く回る順序、小数 6 桁
```

TITLE に埋め込む文字列は互換性のため**サニタイズ**する：ダブルクォート（`"`）と**バックスラッシュ（`\`）**を除去し、印字可能 ASCII 以外（日本語等）は `_` に置換または除去。サニタイズ後に空になる場合は既定名を使用。ダウンロードファイル名自体は日本語可。

**シリアライズ規約**：

- **改行は LF（`\n`）固定**
- 数値は `toFixed(6)`（小数 6 桁固定・**指数表記を排除**）
- 大サイズ（65³ = 274,625 行）のシリアライズは、文字列連結ではなく**配列に push して最後に `join('\n')`** で行い、Worker 側で実行する（メインスレッドをブロックしないため）

## 6. UI 仕様

### 6.0 設計原則

1. **ウィザードにはしない**：画面分割はせず全機能を 1 画面に置く。調整⇄比較を行き来する使い方を妨げないため。操作の誘導はドロップゾーンのガイド文言・点滅ハイライトで行う（進行インジケーターは置かない）
2. **プレビュー常時可視**：パラメーター操作中、結果が必ず画面内に見えていること（本アプリ最重要の UX 要件）
3. **調整は 2 階層**：常時表示は「モード 3 択＋強度・ノイズ抑制スライダー」。露出以下 7 項目（スムージングを含む）は「詳細調整」アコーディオン（既定：閉）に格納し、初見の認知負荷を下げる
4. **書き出し導線は固定**：書き出しバーは常に画面内（デスクトップ：プレビュー直下、モバイル：画面下部 sticky）

### 6.1 ブレークポイントとレイアウト

| 区分 | 幅 | レイアウト |
|---|---|---|
| モバイル | < 768px | 1 カラム縦積み＋2 つの sticky（下記） |
| タブレット | 768–1119px | 2 カラム（左レール 240px ＋ プレビュー） |
| デスクトップ | ≥ 1120px | 2 カラム（左レール 280px ＋ プレビュー）、余白拡大 |

**デスクトップ / タブレット（2 カラム）**

- 左レール（上から）：Source / Reference サムネイル（横並び 2 枚、クリックで差し替え）→ モード 3 択（セグメントコントロール）→ 強度スライダー → 「詳細調整」アコーディオン → リセット。レール自体が縦スクロール
- 右カラム：プレビュータブ（元画像 / 適用後 / 比較〔既定〕）→ 比較スライダー付きプレビュー（`position: sticky` でスクロールに追従）→ 直下に書き出しバー（サイズ選択 / ファイル名 / DL ボタン）
- 参考画像のサムネイルをプレビュー右上に小窓表示（タップで一時拡大）し、「目標」を常に参照できるようにする

**モバイル（1 カラム）**

- 上から：ヘッダー（ロゴ / JA|EN / ?）→ 入力カード 2 枚（横並び）→ **プレビュー（画面上部に sticky）** → モード 3 択 → 強度 → 詳細調整アコーディオン → **書き出しバー（画面下部に sticky）**
- プレビューを上部 sticky にすることで、下のスライダーをスクロール操作している間も結果が見え続ける（原則 2 の実現手段）
- sticky プレビューの高さは画面高の最大 40% にクランプし、スライダー操作領域を確保する。高さ計算には `dvh` 単位または `visualViewport` API を用いる（iOS Safari のアドレスバー可変により `vh` がずれる問題への対策）

### 6.2 状態と遷移

1. **初期（空状態）**：ドロップゾーン 2 つと「サンプル画像で試す」ボタンのみ有効。調整・書き出しはグレーアウト。サンプルボタンは同梱のデモペア（public/）を読み込み、初見ユーザーが 10 秒で全フローを体験できるようにする
2. **片方のみ投入**：もう一方のドロップゾーンを点滅等で誘導
3. **計算中**：プレビュー領域にスケルトン＋プログレス（Worker からの進捗イベント）。UI 操作は可能
4. **生成済み**：全機能有効。パラメーター変更は即時反映、モード変更・画像差し替えは再計算
5. **エラー**：非対応形式・デコード失敗・メモリ不足はトースト表示で通知し、直前の状態を維持する（`>50MP` の事前サイズ検出は廃止し、`createImageBitmap` の失敗を捕捉する方式に統一。§4.1 参照）。HEIC 等の非対応形式では「JPEG に変換してください」等の案内を文言に含める

### 6.3 インタラクション詳細

- **比較スライダー**：デスクトップは境界線上どこでもドラッグ可＋ハンドル。**モバイルはハンドル（直径 44px 以上）のドラッグのみで動作**させ、縦スクロールとのジェスチャー競合を回避する。キーボード（←→）対応
- **スライダー全般**：タッチターゲット 44px 以上。数値ラベルをタップすると直接数値入力できる。ダブルタップ / ダブルクリックで個別リセット
- **プレビュー**：拡大縮小・パン機能は持たない。画像は常にステージへフィット表示する
- **書き出しバー**：DL ボタンは常時 1 つのプライマリボタン。ファイル名・サイズ選択はモバイルではボタン長押し（またはバー内の展開）で表示し、既定値ならワンタップで保存完了

### 6.4 補助 UI・ビジュアル

- ダークテーマ基調（色評価を妨げないニュートラルグレー背景）。UI のアクセント色は彩度低めに抑え、プレビューの色を邪魔しない
- 各スライダーにツールチップで一言説明（i18n 対象）
- `?` ボタンで使い方モーダル（3 ステップ図解＋「LUT が扱える範囲」の注記）
- 「このツールは参考画像の色の雰囲気を移すもの。被写体・明るさが近い 2 枚ほど自然に仕上がり、まったく異なる場面同士では合わない」旨の用途説明ヒントを画像入力の下に常時表示（消去不可）

## 7. 非機能要件

| 項目 | 目標 |
|---|---|
| 自動マッチ計算時間 | 1 秒以内（512px 解析、一般的なノート PC） |

| オフライン | ページを開いたまま（リロードしない限り）はネットワーク不要で全機能動作。完全オフライン対応（Service Worker / PWA）は将来検討 |

```
/
├─ index.html
├─ vite.config.ts
├─ src/
│  ├─ main.ts              # エントリ・UI 結線
│  ├─ ui/                  # ドロップゾーン、スライダー、比較スライダー

│  │  ├─ colorspace.ts     # sRGB↔リニア、Lab 変換
│  │  ├─ mkl.ts            # §5.2
│  │  ├─ histmatch.ts      # §5.3
│  │  ├─ pipeline.ts       # モード合成・§5.5 後段
│  │  ├─ lut.ts            # 格子生成、trilinear 補間、平滑化

│  ├─ gl/preview.ts        # WebGL プレビュー（+ Canvas フォールバック）
│  └─ i18n/{ja,en}.ts
├─ tests/                  # vitest（core/ を対象）
└─ public/                 # サンプル画像（オーナー自身が撮影した実写・権利クリア）
```

**サンプル画像**：`public/` に同梱するデモ用サンプルペアは、**オーナー自身が権利を持つ実写写真**を用いる（第三者の写真・権利未確認の外部素材は使わない）。実写はマッチ品質のデモとして合成画像より説得力があるため。

**Worker とデータ受け渡し（§3 補足）**：

- メイン ⇄ Worker 間のピクセル・LUT データの受け渡しは、**Transferable な `ArrayBuffer`** を用いる（`postMessage` の transfer 指定でコピーを避ける）。`SharedArrayBuffer` は使わない（COOP/COEP ヘッダ設定が不要になり、Cloudflare Pages の静的ホスティングと相性が良いため）
- **プレビュー描画はメインスレッドで行う**（WebGL / Canvas 2D はメインスレッド）。Worker は統計計算・LUT 生成・.cube シリアライズ等の重い純粋計算に限定する

## 11. テスト方針

- **ユニット**：色空間変換の往復誤差、MKL の閉形式解（既知の共分散で検証）、HM の単調性、.cube のフォーマット妥当性
- **ノイズ抑制（§5.3）**：①s=0 で現行実装（σ=2・クランプなし）と完全一致 ②クランプ＋残差平滑化後もカーブが単調 ③CLAHE 式再分配で端点（総上昇量）が保存される ④処理後カーブの最大傾き ≤ S_max(s) ⑤s=100 でカーブが恒等＋定数オフセットに近い ⑥Source＝Reference の同一性がスライダー値によらず維持される
- **ゴールデンテスト**：固定入力ペアに対する LUT 出力のスナップショット比較（アルゴリズム回帰検知）
- **同一性テスト**：Source＝Reference のとき LUT ≒ Identity になること。**スムージング 0・強度 100%** の条件で実施し、全格子点の最大誤差が許容値（例 `1e-3`）以下であることを検証する
- **実写検証**：デジタル写真×フィルム写真のペア数組で目視評価。加えて **iPhone の HDR（ゲインマップ付き）写真**のペアを検証対象に含め、広色域・HDR 由来の入力でも破綻しないことを確認する
- **外部互換**：書き出した .cube を DaVinci Resolve / Photoshop で読み込み、ブラウザプレビューと一致することを確認
